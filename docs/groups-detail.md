# Group Detail UX Specification

## 1. Top Layout

The group detail page is a secondary page inside the Groups navigation flow. Its top area should be native, dynamic, and visually quieter than the content.

### Structure

- The page is pushed from Groups home through a `NavigationStack`.
- Use the system title API with the group name:
  - `.navigationTitle(group.name)`
  - example: `.navigationTitle("May Trip")`
- Use `.navigationBarTitleDisplayMode(.large)`.
- Do not draw a custom title inside the scroll content.
- Do not show a subtitle under the title.
- The first content inside the scroll view starts below the native large title.

### Back Navigation

- Use the system back button generated by `NavigationStack`.
- Do not draw a custom circular back button.
- Do not hide the native back button for this page.
- Preserve the native back swipe gesture.
- The preview should be hosted inside a real navigation path so the back button appears naturally.

### Scroll Behavior

- The title must be native and dynamic.
- At rest near the top, the group name appears as the large navigation title.
- When the user scrolls upward, iOS collapses the large title into the small navigation bar title automatically.
- Do not recreate the collapse behavior with manual geometry, opacity, scale, or overlay text.

### Top-Right Actions

- Show one top-right action: `ellipsis`.
- The `ellipsis` opens `Group settings` directly.
- The settings surface is specified in `docs/group-settings.md`.
- Do not show a separate `Share` text button in the navigation bar.
- Do not mix a text capsule action and an icon action in the top bar.
- Do not wrap top actions in custom glass circles or capsules.

Rules:

- Do not expose `Share` from this entry until sharing is implemented.
- `Archive group` and `Delete group` belong inside `Group settings`, not in the top-level `ellipsis`.
- `Delete group` uses destructive role styling.
- High-frequency content creation does not belong in this menu; adding an expense remains a separate content action.

### Visual Rules

- The navigation bar should inherit the page's warm canvas background.
- Keep toolbar icons visually native and lightweight.
- Avoid custom shadows, borders, or nested glass effects in the toolbar.
- Page-level Liquid Glass may appear in content cards, but not as an extra wrapper around navigation controls.

### Current SwiftUI Pattern

```swift
struct GroupDetailView: View {
    @State private var searchText = ""

    var body: some View {
        ZStack {
            CanvasBackground()

            ScrollView {
                VStack(alignment: .leading, spacing: 16) {
                    SummaryCard()
                    BalancesSection()
                    ExpensesSection()
                }
            }
        }
        .navigationTitle("May Trip")
        .navigationBarTitleDisplayMode(.large)
        .toolbar {
            ToolbarItem(placement: .topBarTrailing) {
                Button {
                    isShowingGroupSettings = true
                } label: {
                    Image(systemName: "ellipsis")
                }
                .accessibilityLabel("Group settings")
            }

            DefaultToolbarItem(kind: .search, placement: .bottomBar)
            ToolbarSpacer(placement: .bottomBar)

            ToolbarItem(placement: .bottomBar) {
                Button {
                    // Add expense
                } label: {
                    Image(systemName: "square.and.pencil")
                }
                .accessibilityLabel("Add expense")
            }
        }
        .searchable(text: $searchText, placement: .toolbar, prompt: "Search")
        .toolbarBackground(.hidden, for: .navigationBar)
    }
}
```

Preview should model the real navigation relationship:

```swift
NavigationStack(path: $path) {
    GroupsHome()
        .navigationDestination(for: Route.self) { route in
            GroupDetailView()
        }
}
```

### Rejected Variants

- Custom in-content header with `Text("May Trip")`.
- Subtitle such as `Hotel, train tickets, dinner` under the title.
- Custom circular back button.
- Standalone `Share` capsule in the top-right toolbar.
- Separate `Share` text action plus `ellipsis` icon action.
- Custom glass styling around toolbar buttons.
- Manually implemented title collapse behavior.

## 2. People Setup Empty State

The people setup empty state appears only for a newly created group that has no other members and no records. It is an invitation to add people, not a blocker.

### Display Condition

Show this state only when both conditions are true:

- The group has exactly one member: the current user.
- The group has no expense records.

Equivalent condition:

```swift
members.count == 1 && records.isEmpty
```

Do not show this empty state when:

- The group has any expense record, even if the current user is the only member.
- The group has any additional member.
- The group is loading; loading state must resolve before empty state is shown.

### Page Structure

- Keep the same native group detail container:
  - `.navigationTitle(group.name)`
  - `.navigationBarTitleDisplayMode(.large)`
  - top-right `ellipsis` for `Group settings`
  - bottom search and add-record toolbar
- Replace the main content sections with the empty state.
- Do not show the summary card.
- Do not show `Balances`.
- Do not show `Expenses`.
- Do not show placeholder cards, fake records, or disabled rows.
- The bottom toolbar remains unchanged; users can still create a record for themselves.

### Content

- Use a small SF Symbol illustration:
  - `person.2`
- Title:
  - `Add people`
- Body:
  - `Add members or temporary members when this becomes a shared expense group.`
- Primary action:
  - `Add people`

The copy must not imply that adding people is required before recording expenses. A one-person group can still be used as a personal ledger.

### Visual Weight

- Match the Groups home empty-state style.
- Use a lightweight centered `VStack`, not `ContentUnavailableView`.
- The icon may sit in a simple circular paper surface with a subtle border.
- Use a transparent native glass button for the action:
  - `.buttonStyle(.glass)`
- Do not use `.borderedProminent` or `.glassProminent` for this empty-state action.
- Do not wrap the empty state in a large card.
- Do not add strong shadows, marketing illustration, onboarding steps, or multiple calls to action.
- Place the block inside the scroll content and visually above the vertical midpoint.

### Interaction

- Tapping `Add people` opens a lightweight People sheet.
- The People sheet contains only:
  - `Add member`
  - `Add temporary member`
- This People sheet is task-focused and separate from `Group settings`.
- `Add member` pushes into the shared member-search flow inside the People sheet stack, with a native trailing chevron.
- `Add temporary member` opens the same lightweight name-entry dialog described in `docs/group-settings.md`.
- The group detail bottom add-record button does not redirect into the people flow.
- The new expense sheet does not show a special "add people first" prompt.

### Current SwiftUI Pattern

```swift
if members.count == 1 && records.isEmpty {
    PeopleSetupEmptyState {
        isShowingPeopleSetup = true
    }
} else {
    SummaryCard()
    BalancesSection()
    ExpensesSection()
}
```

```swift
VStack(spacing: 18) {
    Image(systemName: "person.2")
        .frame(width: 64, height: 64)
        .background(PaperColor(), in: Circle())

    VStack(spacing: 6) {
        Text("Add people")
        Text("Add members or temporary members when this becomes a shared expense group.")
    }

    Button {
        isShowingPeopleSetup = true
    } label: {
        Label("Add people", systemImage: "person.badge.plus")
    }
    .buttonStyle(.glass)
}
```

### Rejected Variants

- Showing the people setup state when the group already has records.
- Blocking `Add expense` until there are at least two members.
- Redirecting the bottom add-record button into member setup.
- Showing an add-people prompt inside the new expense sheet.
- Using `ContentUnavailableView` if it makes the page feel too heavy.
- Showing multiple CTAs such as `Add expense` and `Add people` inside the empty state.

## 3. Summary Area

The summary area explains the user's position in the group. It should be calm and compact, not a full dashboard.

### Primary Content

- Show one summary card directly below the native large title.
- The card's primary value is the current user's net balance in this group.
- Use the label `My balance`.
- Use a prominent, monospaced money value, such as `+¥86.20`.
- Keep the card focused on this personal net balance only.

### What Not To Show

- Do not show `Settle up` inside the summary card.
- Do not show settlement counts or settlement-plan entry points inside the summary card.
- Do not resolve transfers directly from the summary card.
- Do not show `To receive` and `To pay` as separate boxed subcards on the main detail page.
- Do not show member avatars, member count, or record count in the summary card if they duplicate information lower on the page.
- Do not use extra helper text or a subtitle to fill space.

### Visual Rules

- The card should feel important but not heavy.
- Use a simple card surface with a subtle border.
- Light shadow is acceptable only when it supports depth; avoid a floating, promotional-card feel.
- Keep the card shorter than the early B variant: the summary should not push `Balances` too far down the page.

## 4. Balances

Balances gives the user a quick explanation of who they owe or who owes them, without turning the main page into a settlement workflow.

### Main Page Section

- The section title is `Balances`.
- Do not expose `Settle up` as a persistent action on the main detail page.
- The main page shows a preview of balances, not the full settlement workspace.
- Show up to `3` balance rows by default.
- Do not include the current user as a row; the summary card already represents the current user's balance.
- Each row represents a relationship between the current user and another member.

### Row Content

Each preview row should include:

- Member avatar or initial.
- Member name.
- Short record summary, such as `3 records`.
- Signed amount.
- Disclosure indicator.

Rules:

- Do not show text such as `owes me` or `I owe` as a secondary label in the row; sign and color carry that meaning.
- Positive values use the positive color.
- Negative values use the negative color.
- Settled or zero values use a neutral color.
- Money values must remain readable and should use monospaced digits.

### Detail Entry Row

The bottom row of the balances card is always the entry to the full balances sheet.

Copy rules:

- If there are more than `3` balances, use `View all`.
- If there are `3` or fewer balances, use `View details`.
- Do not use `Show more`; it sounds like list expansion rather than entering a complete balance workspace.
- Do not put an icon before this row.
- Use text plus a trailing disclosure indicator only.

### Full Balances Sheet

Opening the balances entry presents a native sheet, not a pushed navigation page.

Presentation:

- Use `.sheet`.
- Use medium and large detents:
  - `.presentationDetents([.medium, .large])`
- Show the drag indicator:
  - `.presentationDragIndicator(.visible)`
- The sheet starts as a half-height layer and can be pulled higher.
- Closing the sheet returns to the group detail page without losing scroll position.

Structure:

- The sheet owns a local `NavigationStack`.
- The title is `Balances`.
- Do not add a `Done` or `Close` toolbar button; the visible drag indicator and sheet gesture are the close affordance.
- Show the full balances list.
- Do not repeat an `All balances` heading directly under the sheet title.
- Include settlement context or a settlement plan inside the sheet, because the user has explicitly entered the balances workspace.

### Member Balance Detail

A member balance detail explains why a specific person has a balance with the current user. It lives inside the balances sheet workspace.

Entry points:

- Tapping a member row in the main page `Balances` preview opens the balances sheet and immediately navigates to that member detail.
- Tapping `View all` or `View details` opens the balances sheet root.
- From the balances sheet root, tapping a member row pushes that member detail inside the sheet's local `NavigationStack`.

Navigation rules:

- Do not present member balance detail as another half sheet.
- Do not push member balance detail onto the parent group-detail navigation stack.
- Use the balances sheet's local `NavigationStack` so the user can go back to the full balances list.
- Closing the sheet from any level returns to the group detail page.

Content:

- Use the member name as the native navigation title, such as `Lin`.
- Show a compact summary with:
  - balance with that member
  - record count
- Do not repeat the group-level `My balance` card.
- Do not repeat the full settlement plan on every member detail unless the plan is scoped to that member.
- Show records related to that member, including records where:
  - the member paid
  - the member was included in a payment made by someone else

Record row rules:

- Use a lighter version of the main `Expenses` row.
- Keep the row tappable with a disclosure indicator.
- Include:
  - category icon
  - record title
  - payer/member relationship summary
  - amount
  - date or time
- Example meta copy:
  - `@Lin`
  - `@Hong`
- Keep category icons smaller and quieter than the main expense list.

Editing flow:

- Tapping a record opens the shared `Edit expense` sheet.
- The edit sheet is presented above the balances workspace.
- Saving or cancelling the edit sheet returns to the same member balance detail.
- After saving, refresh the member detail and the balances root data without dismissing the balances sheet.

### Settlement Flow

- Settlement belongs inside the balances sheet, not as a permanent main-page CTA.
- The sheet may include a suggested settlement plan.
- Do not show a repeated `Suggested settlement` subtitle on every transfer row.
- Do not show a separate `2 steps` count beside the section heading; the primary CTA already communicates the transfer count.
- Settlement actions should feel contextual and reversible where possible.
- If settlement becomes a multi-step process, it can continue inside the sheet with local navigation or confirmation, rather than pushing the whole group detail page away.

### Sheet Bottom Action

The balances sheet can expose one settlement CTA because the user has explicitly entered the balances workspace.

Placement:

- Keep the action floating at the bottom of the sheet content.
- Do not place this text CTA in `.toolbar(placement: .bottomBar)`; toolbar overflow can collapse it into an ellipsis.
- Do not place it in `safeAreaBar` if the result becomes a flat full-width bar detached from content.
- The button should sit above content enough to feel tappable, but it should not cover important rows at rest.

Visual style:

- Use the native system button style:
  - `.buttonStyle(.glassProminent)`
  - `.controlSize(.large)`
- Do not add custom blur, custom capsule backgrounds, custom strokes, or tint.
- Do not use `.glass(.clear)` for this primary settlement action; it is too subtle on the warm sheet background.
- Do not use a hand-built `glassEffect` capsule for the CTA unless the system button style proves unusable in production.
- The label should be action-specific and count-aware, such as `Resolve 2 transfers`.

### Current SwiftUI Pattern

```swift
struct GroupDetailView: View {
    @State private var isShowingBalances = false

    var body: some View {
        ScrollView {
            VStack(alignment: .leading, spacing: 16) {
                SummaryCard()
                BalancesSection {
                    isShowingBalances = true
                }
                ExpensesSection()
            }
        }
        .sheet(isPresented: $isShowingBalances) {
            NavigationStack {
                BalancesSheet()
            }
            .presentationDetents([.medium, .large])
            .presentationDragIndicator(.visible)
        }
    }
}

struct BalancesSheet: View {
    var body: some View {
        ZStack(alignment: .bottom) {
            ScrollView {
                VStack(alignment: .leading, spacing: 16) {
                    AllBalancesList()
                    SettlementPlan()
                }
            }

            Button("Resolve 2 transfers") {
                // Start settlement flow
            }
            .buttonStyle(.glassProminent)
            .controlSize(.large)
            .accessibilityLabel("Resolve 2 transfers")
        }
        .navigationTitle("Balances")
        .navigationBarTitleDisplayMode(.inline)
    }
}
```

## 5. Expenses

Expenses is the chronological activity list for the group.

### Section Naming

- Use `Expenses`.
- Do not use `Recent expenses`; it is wordier and implies a separate archive of older expenses before that behavior exists.

### Role

- Keep expenses on the main group detail page.
- Do not move expenses into a secondary panel.
- The section should remain visually lighter than the summary card but still clearly scannable.
- Do not show a right-side total count in the section header, such as `12 total`; it adds noise without helping the main scan.

### Row Content

Each row should include:

- Expense title.
- Payer and participant count.
- Amount.
- Time or date.
- Disclosure indicator.

Rules:

- Category icons are acceptable for expenses because they describe the expense itself.
- Use compact rows optimized for scanning.
- Keep amounts aligned and monospaced.
- Do not let floating add controls obscure important row content.

## 6. Add Expense Action

Adding an expense is a high-frequency content action. It should remain visible, but it should use the native bottom toolbar instead of a custom floating button.

### Bottom Toolbar Structure

- Use the iOS 26 native bottom toolbar pattern.
- Put search on the left using the system search toolbar item.
- Put add expense on the right as a separate toolbar item.
- Use `ToolbarSpacer(placement: .bottomBar)` between them so the system lays them out as two distinct controls.
- Do not draw a custom floating `+` button over the content.
- Do not build a custom bottom glass bar for this page.

### Search

- Search is available from the bottom toolbar.
- Use `DefaultToolbarItem(kind: .search, placement: .bottomBar)`.
- Pair it with `.searchable(...)` so the system owns the search field, prompt, focus behavior, keyboard handling, and expansion.
- Use the prompt `Search`.
- Search should filter expenses and/or group-local content when wired to real data.
- Do not use a top navigation search field for this page.

### Add Expense

- Use a native `Button` in `.bottomBar`.
- Use the compose-style SF Symbol:
  - `square.and.pencil`
- Accessibility label: `Add expense`.
- Avoid the plain `plus` symbol here; in this context, compose better communicates creating a new expense record.
- Do not use a custom circular background, manual glass effect, or manual safe-area positioning.

### Current SwiftUI Pattern

```swift
@State private var searchText = ""

// ...

.toolbar {
    DefaultToolbarItem(kind: .search, placement: .bottomBar)
    ToolbarSpacer(placement: .bottomBar)

    ToolbarItem(placement: .bottomBar) {
        Button {
            // Add expense
        } label: {
            Image(systemName: "square.and.pencil")
        }
        .accessibilityLabel("Add expense")
    }
}
.searchable(text: $searchText, placement: .toolbar, prompt: "Search")
```

### Rejected Variants

- Custom floating `+` button.
- Custom glass capsule containing search and add.
- Top-only search.
- One combined toolbar item that visually merges search and add.
- Plain `plus` icon for the expense creation action.

## 7. Expense Editor Sheet

The expense editor is used for both creating and editing an expense. It is a short, focused data-entry task, so it should stay in a native sheet instead of replacing the group detail page.

### Presentation

- Present the editor with `.sheet`.
- Use a local `NavigationStack` inside the sheet.
- Use the large detent by default:
  - `.presentationDetents([.large])`
- Show the drag indicator:
  - `.presentationDragIndicator(.visible)`
- Do not first present a medium sheet and then expand automatically.
- Closing or saving returns to the group detail page without changing the parent scroll position.

### Navigation And Titles

- New expense title: `New expense`.
- Existing expense title: `Edit expense`.
- Add and edit should reuse the same form structure.
- Tapping an expense row opens `Edit expense` in this sheet.
- Do not push the editor into the main group-detail navigation stack.
- Use native toolbar placements:
  - cancellation action on the left
  - confirmation action on the right
- The cancel action uses an `xmark`.
- The save action uses a `checkmark`.
- The save action is disabled until the minimum required input is valid.

### Accent Color

- Apply the page theme color to the editor root with `.tint(theme.accent)`.
- This tint should affect native controls such as:
  - text insertion cursor
  - menu picker value
  - compact date picker active state
  - toolbar confirmation action
- Prefer local tint on the editor first. Use the app-level `AccentColor` asset only when the same accent should apply globally.
- Do not manually recolor native picker internals with custom overlays.

### Field Priority

The editor should support one-screen completion for common expenses.

Primary fields:

- Amount
- Paid by
- Split
- Category

Secondary fields:

- Date
- Optional note

Rules:

- Do not show `What was it for?` as a separate required or optional title field.
- Do not duplicate the same concept with both `What was it for?` and `Optional note`.
- Do not include `Receipt` in the primary editor until there is a real receipt workflow.
- Do not put `Date` and `Optional note` into a separate `More details` section; keep them in the same form card so the sheet remains compact.

### Defaults

- Amount may be prefilled in the design playground for visual review, but production should start empty unless launched from a pre-parsed source.
- Paid by defaults to the current user.
- Split defaults to no selected members.
- Category defaults to the most common or last-used category when available; otherwise use a neutral default such as `Meal`.
- Date defaults to the current date and time.
- Optional note defaults to empty.

Save eligibility:

- Save requires a valid positive amount.
- If production validation requires at least one split member, communicate that inline and disable save until the condition is met.
- Do not require a title or note to save.

### Amount

- Amount is the most important field and appears alone in the first section.
- Use a large numeric text field with decimal keyboard.
- Use monospaced digits.
- Keep the currency symbol visible but visually secondary.
- The text insertion cursor should use the sheet tint.

### Paid by

- `Paid by` is a high-frequency short choice.
- Use a native `Picker` with `.pickerStyle(.menu)`.
- Do not navigate to a separate picker page for normal member counts.
- The selected payer value appears on the trailing side of the row.
- The menu value and indicator should use the sheet tint.

### Split

- `Split` is a high-frequency field and should be editable inline.
- Do not push to a secondary split page for the default equal-split case.
- Show a compact member-selection grid under the `Split` label.
- Each member item shows:
  - circular initial avatar
  - one-line name label
- Selected members use the theme accent fill.
- Unselected members use a quiet neutral surface.
- Long member names are single-line and tail-truncated.
- The default state is no selected members.
- Provide an `All` shortcut in the `Split` header row.
- `All` is a toggle:
  - if not all members are selected, tapping it selects all
  - if all members are selected, tapping it clears all
- Do not show helper text such as `Equally · 10 people`; it repeats information already visible in the grid.

Layout:

- The grid should behave like a justified row layout.
- Compute how many fixed-width items fit in the available row width.
- Distribute remaining horizontal space as row gaps.
- Do not use a fixed left-aligned grid that leaves a large empty area on the right.
- Do not use flexible columns that stretch item gaps unpredictably.
- The same layout should adapt across sheet widths.

### Category

- `Category` is a high-frequency visual choice.
- Show category choices inline as a compact icon palette.
- Do not show a separate trailing text value for the currently selected category; the selected icon state is enough.
- Do not push to a secondary category page for the default category set.
- Category items use the same visual language as expense rows:
  - circular icon container
  - SF Symbol icon
  - short label
- Selected category uses a clear accent indication, such as an accent stroke around the icon.
- Keep labels one line and tail-truncated.

Default category set should cover the existing app categories:

- Meal
- Drink
- Hotel
- Shopping
- Transport
- Stay
- Vacation
- Transfer
- Ticket
- Game
- Other

### Date

- Use a native compact `DatePicker`.
- Keep it in the same form card as the other metadata.
- The date defaults to now.
- The compact control should keep native behavior; do not custom draw the date capsules.

### Optional note

- Use a multiline `TextField` with vertical axis.
- Label/copy: `Optional note`.
- Keep it optional.
- It should not be required for save.

### Delete expense

- Delete is only available in `Edit expense`.
- Add one independent destructive entry at the bottom of the editor form.
- Use native destructive styling, such as `Button(role: .destructive)`.
- Label: `Delete expense`.
- Do not place delete in the top toolbar.
- Do not show delete in `New expense`.
- In production, destructive deletion should ask for confirmation before committing.

### Advanced Split

- The inline split grid covers the default equal-split case.
- If the user needs exact amounts, shares, or weighted split, use a secondary split editor from the split area.
- The advanced split editor should remain inside the expense editor sheet's local navigation stack.
- Do not expose advanced split controls by default on the main editor form.

### Rejected Variants

- Pushing the editor as a full detail page in the group navigation stack.
- Presenting a half sheet first and expanding automatically.
- Putting `Paid by`, `Split`, and `Category` behind separate drill-down pages for the common case.
- Requiring a title or `What was it for?` field.
- Showing both `What was it for?` and `Optional note`.
- Splitting date and note into a separate `More details` section.
- Showing Receipt before receipt capture exists.
- Default-selecting all split members.
- Custom-drawing native picker/date controls just to change their accent color.
